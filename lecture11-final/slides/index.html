Getting Familiar with Linux/Command-Line/Bash Scripting/Server Administration

  • Linux has substantial marketshare and is used by many major tech companies (Google, Amazon, etc)
  • Most flavors of Linux are free/open-source: anyone can download and use the free Linux distros
  • The Git-Bash that I asked you to install emulates the Linux command line environment. You can run most Linux commands/scripts directly in Git Bash. Likewise for OSX terminals. Deploying to CS3 was also practice.
  • There are many free resources and vidoes online to learn on this topic. In addition to online resources, I recommend the following two books (my personal favorites)
  • The Linux Command Line: https://www.amazon.com/Linux-Command-Line-Complete-Introduction/dp/1593273894
  • UNIX and Linux System Administration Handbook: https://www.amazon.com/UNIX-Linux-System-Administration-Handbook/dp/0131480057
  • The best way to practice this is to try deploying a web application to a Linux Server on a public hosting service. Often times learning new technology is driven by the need to do so.

Deploying Your Web Application for the Public

  1. Address security issues:
    Assume everything in the browser can be comporised and automated by malicious bots. Be careful on how you handle authentication and make sure to validate user input on the server side before anyt interaction with the database. If the server host charges by usage, make sure you limit public usage by using some sort of authentication.
  2. Find a public host:
    I recommend DigitalOcean as a good starting point. The GitHub student pack gives you 50 dollar credit for DigitalOcean. Some other choices are Heroku and AWS
  3. Read the documentation provided by the public host on how to deploy, and deploy the applicaiton. It might just be a vanilla linux server so you would google "how to run java app in linux" etc.
  4. Access your deployed public web app, probably using the server ip address plus whatever resource path/port your defined. You might have to open ports.
  5. Optional: register a domain (website address) using Google Domains, GoDaddy, etc. Read the domain provide and public host documentation on how to set up the Host Name.

Learning About Distributed Systems

  • Many web applications (Google, Amazon, Uber, etc.) have global presence and requirements on reiliability and fault-tolerance. This is a very important interview topic for full-time positions
  • They also have many, many users and many, many servers.
  • How do you handle server failures? How do you handle traffic spikes (cybermonday shopping)? How do you handle latency between servers that are in different locations around the globe? How do you write code that accounts for those factors?
  • Distributed Systems Reading List: https://christophermeiklejohn.com/distributed/systems/2013/07/12/readings-in-distributed-systems.html
  • Distributed Systems for Young Bloods: https://www.somethingsimilar.com/2013/01/14/notes-on-distributed-systems-for-young-bloods/
  • How We Scaled Dropbox
  • CS436 Distributed Computer Systems (Univeristy of Waterloo): https://cs.uwaterloo.ca/~rtholmes/teaching/2011winter/cs436/

Internship/Full-Time Interview Preparation

  • Understand the type of interview the company does. Do some research online via glassdoor/quora/careercup
  • Don't forget to study for behavorial questions (projects you worked on / problem you faced/solved)
  • "Big 4" type companies technical interview:
    • Heavy on data structures and algorithms
    • Pre-screening: Solve problems online or over the phone
    • On-site: Solve problems on the whiteboard. Internship usually one or two. Full time can be four or more in a day.
  • Startups Technical Interview:
    • Can require hands-on coding challenges in addition to data structures/algorithms
    • Take-home or onsite programming assignments
    • For example: design and implement a basic shopping website with the following requirements
    • They want to see that you are immediately useful when you start. Coding school graduates have the edge on this one.
